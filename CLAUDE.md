# 더핀 인트라넷 작업 규칙

> 이 파일은 Claude Code가 세션 시작 시 자동으로 읽는 프로젝트 컨텍스트·작업 규칙 통합 파일입니다.
> 사용자 지정 작업 규칙이 상단(§1~§10), 프로젝트 상세는 하단(§11) 영역에 있습니다.

---

## §1. 회사 컨텍스트

- 더핀중개법인: 경공매 전문 부동산 법인, 서울 기반
- 7단계 워크플로우: 블로그 홍보 → 상담 → 물건조사 → 보고서 → 입찰동행 → 대출연계 → 명도
- 수수료: 아파트 1% / 다가구·건물·공장·토지 2% / 특수물건 3%
- 매니저: 손청락(대표), 이제훈(이사·명도), 장한빛, 정은진
- 고객 1명당 매니저 1명 + AI 에이전트 1개 배정 구조
- 인프라: Netlify(index.html) + GAS(unified-gas.js) + Telegram 봇(@thefin1_bot) + Google Sheets

---

## §2. 작업 원칙

- 모든 작업은 **ultrathink** 수준으로 분석 후 시작
- **질문 금지.** 모호한 결정은 합리적 가정으로 진행하고, 가정은 마지막 보고에 명시
- 작업 시작 시 **TaskCreate로 단계별 계획 작성**, 단계마다 status 갱신
- 사람 개입 없이 Definition of Done 도달 못하면 끝난 게 아님

---

## §3. Definition of Done (공통)

1. 기능 실제 동작 (브라우저 콘솔/네트워크 에러 0개)
2. 기존 동작 회귀 없음
3. 기존 코드 스타일·네이밍 유지
4. 변경 요지를 주석/README에 반영

---

## §4. 검증 루프 (실패 시 최대 3회 자가 수정)

1. **정적 검사**: 문법 에러, 미사용 변수, 디버그 콘솔 잔재
2. **실행 검증**:
   - 프런트엔드 → 로컬에서 실제 렌더 확인 (가능하면 스크린샷)
   - GAS → curl로 webhook 호출 후 응답 검증
   - Telegram 봇(@thefin1_bot) → 봇 명령 시뮬레이션
3. **회귀 검증**: 변경 파일과 관련된 기존 기능 1~2개 추가 점검

---

## §5. Self-Review 체크리스트 (구현 후 별도 단계로 강제)

- [ ] 엣지케이스 (빈 입력, 네트워크 실패, 권한 없음, 한글 깨짐, 모바일 화면)
- [ ] 보안 (API 키 노출, XSS, 입력 검증)
- [ ] 성능 (불필요한 리렌더, N+1 호출, 큰 이미지)
- [ ] UX (로딩 표시, 에러 메시지, 되돌리기)
- [ ] 명세 누락 항목

## §6. 멀티 페르소나 협업 모드

### 6.1 트리거 조건
다음 중 하나면 멀티 페르소나 모드 진입:
- 새 기능 추가 (UI + 데이터 + 외부 연동 중 2개 이상)
- 기존 워크플로우 변경
- 법적·세무적·금융적 함의 있는 기능
- 50줄 이상 변경

### 6.2 스킵 조건 (dev 페르소나만 호출)
- 단순 버그 수정
- 문구·라벨 변경
- 스타일·색상 조정
- 30줄 이하 변경

### 6.3 9 직군 (`.claude/agents/`)
| 직군 | 정상 | Devil's Advocate |
|---|---|---|
| 개발 | `dev` | `devils-advocate-dev` |
| 영업·운영 컨설팅 | `consultant` | `devils-advocate-consultant` |
| 제품 관리 | `pm` | `devils-advocate-pm` |
| 법률 | `lawyer` | `devils-advocate-lawyer` |
| 세무 | `tax` | `devils-advocate-tax` |
| 부동산 평가 | `appraiser` | `devils-advocate-appraiser` |
| 지식재산권 | `ip` | `devils-advocate-ip` |
| 대출 컨설팅 | `loan` | `devils-advocate-loan` |
| 시장 리서치 | `researcher` | `devils-advocate-researcher` |

### 6.4 Round 1 — 직군별 의견 수렴
- Agent 도구로 관련 직군 subagent **병렬** 호출
- `consultant`·`pm`·`dev` 3직군은 거의 항상 포함
- 각 직군 의견을 **쟁점 리스트**로 정리

### 6.5 Round 2 — 쟁점별 깊이 토론
쟁점 1개당:
1. 해당 직군 정상 에이전트 호출
2. 해당 직군 Devil's Advocate 호출
3. 두 의견을 비교해 **trade-off 명시**
- "같은 직군 3명" 패턴 = [정상 + Devil's Advocate + 메인 Claude의 최종 판단] 으로 구현

### 6.6 Round 3 — 통합 구현
1. 모든 피드백을 요구사항으로 통합
2. `dev` 에이전트(또는 `fin-dev`)가 설계·구현
3. **검증 루프 + Self-Review** 강제 실행
4. **별도 dev 에이전트 Task로 코드 리뷰** (50줄+ 변경 시)

### 6.7 충돌 해결 우선순위
- 법적·세무적 리스크 → 보수적 입장 채택
- UX vs 구현 난이도 → consultant 우선
- 성능 vs 기능 → pm 우선
- 그 외 → 메인 Claude 판단 + **이유를 ADR에 기록**

## §7. 의사결정 로그 (ADR)

중요한 결정은 **`docs/decisions/ADR-YYYYMMDD-제목.md`** 로 저장
포맷: **컨텍스트 / 결정 / 근거 / 대안 / 영향**
템플릿과 작성 규칙은 `docs/decisions/README.md` 참조.

다음 경우는 ADR 작성 의무:
- 멀티 페르소나 모드 충돌 해결 결과
- 스키마 변경
- 외부 서비스(API·금융기관·법무) 연계 결정
- 6개월 후 누군가 "왜 이렇게 짰지" 물을 만한 모든 것

## §8. 금지 사항

- `rm -rf`, `git push --force`, GAS 자동 배포 명령 금지
- API 키·환경변수 코드 하드코딩 금지
- 시트/컬럼 스키마 변경 시 **마이그레이션 영향 반드시 보고**
- main 브랜치에 미완성 코드 직접 push 금지
- 고객 연락처·주민번호는 로그·Git에 절대 남기지 않음

## §9. Subagent 활용

- **50줄 이상 변경** 또는 **신규 기능 추가** 시 → Agent 도구로 별도 리뷰어 띄워 셀프 리뷰
- 리뷰 피드백은 별도 커밋으로 반영 (원 작업과 분리하여 추적 가능)

# §11. 프로젝트 상세

## §11.1 프로젝트 개요

**THE FIN(더핀)의 자체 인트라넷 및 웹 플랫폼.**
2단계로 나누어 개발한다.

### Phase 1 (현재 단계) — 직원 업무 효율화 인트라넷
- 대표·이사가 직원에게 업무 지시 + 진행 상태 한눈에
- 고객/사건 단위 7단계 워크플로우 추적
- 담당자별 할당 현황 실시간
- 블로그 4개 계정 게시 현황 통합 관리

### Phase 2 (차후) — 고객 편의성 기능
- 고객 자가 조회 (본인 사건 진행 상태)
- 온라인 상담·문의 접수
- 공개 경매 물건 페이지

**원칙: Phase 1 안정화 전까지 Phase 2 기능은 추가하지 않는다.**

## §11.3 기술 스택

**프런트엔드:** `index.html` 단일 파일 SPA (~17,000줄) — Vanilla HTML/CSS/JavaScript (프레임워크 없음)
**백엔드:** `unified-gas.js` — Google Apps Script (~2,500줄), Google Sheets DB
**외부 API:** Claude API (AG_ OCR), Telegram(@thefin1_bot)
**인증:** 커스텀 RBAC — `AUTH_login()` + `localStorage` + 비밀번호 해싱
**빌드 시스템:** 없음. Netlify로 정적 호스팅. **빌드 도구·프레임워크 도입 금지.**

## §11.4 명령어

```bash
node scripts/validate.js                                 # 검증 (수정 후 필수)
git worktree add ../fin-feature-<기능명> feature/<기능명>  # 새 기능 시작
git checkout main && git merge --no-ff feature/<기능명>   # 완성 후 머지
git worktree remove ../fin-feature-<기능명>
git branch -d feature/<기능명>
```

`validate.js`는 다음 7가지를 검사: 파일 크기(60KB 이상), 필수 DOM ID 12개, 필수 함수 8개, TDZ 안전성, 하드코딩된 시크릿 없음, GAS URL 참조 유지, 브랜드 색상 변수 사용.
`.claude/settings.json`의 PostToolUse 훅이 Edit/Write 도구 사용 후 자동 실행.

## §11.5 SPA 아키텍처

### 패널 탭 & 모듈 접두어
| 탭 | 접두어 | 주요 기능 |
|---|---|---|
| 사건 관리 | `C_` | 사건 CRUD, 7단계 상태 |
| 경매 일정 | `A_` | 입찰 기록, 명도 자동 연결 |
| 명도 | `M_` | 명도 진행 타임라인 |
| 캘린더 | `CAL_` | 시각적 경매 일정, ICS 내보내기 |
| 대시보드 | `DASH_`, `CD_`, `FEE_` | 워크로드·수수료 |
| 블로그 초안 | `DRAFT_` | 자동 블로그 스크립트 |
| 입찰 안내 | `AG_` | 경매 안내 카드, 사진 OCR |
| 메시지 | `MSG_` | 직원 메시지, 오류 제보 |
| 회원 관리 | `MBR_` | 고객 정보 |
| 약정서 | `CTR_`, `CONTRACT_` | 계약 보관 |
| 소송 | `LW_` | 소송 진행 사건 |

### 페이지 로드 순서
```
AUTH_login() → AUTH_applyRole() → loadBulk() → applyBulkData() → 각 패널 render()
```

### 데이터 흐름
```
UI 변경 → gasPost('update') → GAS가 Sheets 업데이트
       → C_render() + A_render() 재렌더링
       → localStorage 섀도 캐시 (FEE_SHADOW) 동기화
```

### 섀도 스토리지 (`thefin_fee_shadow_v1`)
GAS 백엔드에 없는 프런트 전용 필드(bidPrice, currentStage, staffShares, manualFee)는 localStorage에. `FEE_migrate()`가 로드 시 복원.

### GAS 라우팅
```
doGet/doPost → module 파라미터로 분기:
  'blog' / 'auction' / 'myeongdo' / (기본=cases)
```

## §11.9 개발 워크플로우 — 워크트리 기반

```
1. 워크트리: git worktree add ../fin-feature-<기능명> feature/<기능명>
2. 워크트리 안에서 개발 (index.html, unified-gas.js 수정)
3. 완성 형태로 만든 후에만 머지 (테스트 계정으로 검증)
4. main에 머지 & 워크트리 정리
5. git push (사용자 명시 승인 필요)
```

규칙:
- main 브랜치엔 항상 동작하는 완성 코드만
- 작업 중 코드는 feature 브랜치에서만 커밋
- 머지 전 테스트 계정(테스트매니저·테스트이사)으로 검증
- 워크트리 디렉토리명: `../fin-feature-<기능명>` (프로젝트 폴더 밖)

### GAS 배포 (수동)
GAS 수정 시: Apps Script 에디터에서 `unified-gas.js` 붙여넣고 새 버전 배포. Script Properties에 `ANTHROPIC_KEY`, `TELEGRAM_TOKEN` 설정 필요. **Claude Code는 배포 자체 시도 금지 — 체크리스트만 제공.**

## §11.11 반드시 지킬 것
