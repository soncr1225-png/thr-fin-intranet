## 📌 프로젝트 개요

**THE FIN(더핀)의 자체 인트라넷 및 웹 플랫폼.**
2단계로 나누어 개발한다.

### Phase 1 (현재 단계) — 직원 업무 효율화 인트라넷
- 대표·이사가 직원에게 업무를 지시하고 진행 상태를 한눈에 확인
- 고객/사건 단위로 7단계 워크플로우 추적
- 담당자별 할당 현황 실시간 파악
- 블로그 4개 계정 게시 현황 통합 관리

### Phase 2 (차후) — 고객 편의성 기능
- 고객 자가 조회 (본인 사건의 진행 상태)
- 온라인 상담 및 문의 접수
- 공개 경매 물건 페이지

**원칙: Phase 1이 안정화되기 전까지 Phase 2 기능은 추가하지 않는다.** 내부 도구가 먼저, 고객 대면은 나중.

## 🏢 비즈니스 컨텍스트

THE FIN은 **부동산 경매·공매 전문 중개법인**이며, 장기적으로 종합부동산 서비스로 확장을 목표로 한다.

### 7단계 핵심 업무 흐름
모든 사건(고객)은 아래 7단계 중 하나의 상태를 가진다. 인트라넷의 중심 데이터 모델은 이 흐름을 반영한다.

1. **블로그 홍보** — 매니저 (4개 계정에 게시)
2. **상담** — 대표/이사
3. **조사 요청** — 매니저가 수행
4. **보고서** — 매니저 작성
5. **입찰 동행** — 대표/이사
6. **대출 연결** — 대표/이사
7. **명도** — 이이훈 이사 전담

### 수수료 구조
| 물건 유형 | 요율 |
|---|---|
| 아파트 | 1% |
| 다가구 · 빌딩 · 공장 · 토지 | 2% |
| 특수물건 | 3% |

직원 배분제 운영 (상세 비율은 내부 정책 참고).

### 고객 담당 체계
- 고객 1인당 담당자 1명 배정 (현재)
- 추후 AI 1명 추가 배정 예정 → **2인 1조(사람 + AI) 구조**

## 🛠 기술 스택 (미확정 — 시작점)

> 아직 확정되지 않음. 아래는 출발점이며, 변경 시 이 섹션을 **먼저** 업데이트한다.

**추천 스택 A — 범용 웹앱 (확장성 우선)**
- 프런트엔드: Next.js (React + TypeScript)
- 스타일링: Tailwind CSS
- 백엔드/DB: Supabase (PostgreSQL + Auth + Realtime)
- 호스팅: Vercel

**추천 스택 B — Google 생태계 통합 (기존 자산 재활용 우선)**
- AppSheet 또는 GAS 기반 웹앱
- Google Sheets를 데이터 소스로 유지
- 기존 Telegram 봇·GAS 자동화와 자연스럽게 연결

**현재 판단:** 기존 자산(Sheets, GAS, Telegram 봇)이 이미 가동 중이므로 **단기적으로는 B, 장기적으로는 A로 마이그레이션** 하는 경로가 현실적. Claude Code는 이 판단을 대표와 다시 확인한 후 구현 시작.

---

## 🧱 기존 자산 (재개발 금지, 점진적 통합)

- **경매 입찰가 산출 Excel 워크북** (6개 시트, 브랜드 적용 완료)
  - 입찰가격표 시트: 3존 시각 체계 (레드/블루)
- **경매 물건 추적 Google 스프레드시트**
  - 사건번호, 물건정보, 매각기일, 블로그 4개 계정 게시 현황, 추천 키워드
  - GAS 자동화 진행 중 (디버깅 단계)
- **Telegram 봇 + GAS + Google Sheets** 연동
  - 사진 1장 입력 기반 자동화 (대법원 API 대체)
- **블로그 4개 계정** (계정명은 내부 문서)

인트라넷은 위 데이터와 **동일한 소스를 바라보거나**, 점진적으로 마이그레이션한다. **기존 파일을 덮어쓰거나 구조를 깨뜨리지 말 것.**

## 🧭 Phase 1 개발 우선순위

1. **사건 대시보드** — 전체 사건을 7단계별로 시각화 (칸반 또는 테이블 뷰)
2. **담당자별 워크로드 뷰** — 누가 어떤 건을 얼마나 맡고 있는지
3. **업무 지시/진행 체크** — 대표·이사가 매니저에게 태스크 할당, 매니저가 완료 처리
4. **블로그 4개 계정 게시 현황 통합 뷰** — 기존 스프레드시트와 양방향 동기화
5. **매각기일 캘린더 + 알림** — D-7, D-3, D-1 텔레그램/이메일 알림
6. **보고서 템플릿 자동 생성** — 조사 데이터 입력 → PDF/DOCX 출력

**1~5번까지가 Phase 1 MVP.** 6번부터는 확장 기능.

## ✅ 코딩 · 운영 컨벤션

- **언어:** TypeScript 기본, `any` 타입 지양
- **시간대:** 모든 timestamp는 Asia/Seoul (KST)
- **금액:** 원 단위, 천 단위 콤마 (`1,250,000원`)
- **UI 텍스트:** 전부 한국어
- **커밋 메시지:** 한국어 가능, `feat:` `fix:` `chore:` `docs:` 접두어 사용
- **파일명:** kebab-case (`case-detail-page.tsx`)
- **민감 정보:** 고객 연락처·주민번호 등은 **절대 로그·Git에 남기지 않는다**

## ⚠️ Claude가 반드시 지킬 것

0. **파일 읽기·수정·생성·검색 등 일반적인 코딩 작업은 승인 없이 즉시 실행한다.** 매번 확인을 구하지 말고 오토 모드로 진행할 것. 단, 파일 삭제·git push·외부 서비스 호출처럼 되돌리기 어려운 작업은 예외.
1. **파일 수정 전 `git status` 확인.** 미커밋 변경이 있으면 먼저 커밋하거나 대표에게 확인.
2. **DB 스키마 변경은 반드시 마이그레이션 파일로 관리.** 직접 DB 수정 금지.
3. **Phase 2 기능을 Phase 1에 섞지 말 것.** 고객 대면 기능은 별도 브랜치/이슈로 분리.
4. **기존 Excel / Google Sheets 파일을 덮어쓰지 말 것.** 원본은 항상 읽기 전용으로 취급.
5. **대법원 API는 사용 불가.** 데이터는 수동 입력 또는 사진 1장 기반 자동화로만 유입.
6. **법률 자문 형태의 응답 금지.** 업무 지원과 시스템 구현에 한정.
7. **복잡한 작업은 먼저 계획(Plan)을 제시하고 대표의 승인 후 실행.** 바로 코딩 시작 금지.
8. 불확실한 결정은 **추측하지 말고 대표에게 질문**.
