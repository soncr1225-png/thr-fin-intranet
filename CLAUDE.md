## 📌 프로젝트 개요

**THE FIN(더핀)의 자체 인트라넷 및 웹 플랫폼.**
2단계로 나누어 개발한다.

### Phase 1 (현재 단계) — 직원 업무 효율화 인트라넷
- 대표·이사가 직원에게 업무를 지시하고 진행 상태를 한눈에 확인
- 고객/사건 단위로 7단계 워크플로우 추적
- 담당자별 할당 현황 실시간 파악
- 블로그 4개 계정 게시 현황 통합 관리

### Phase 2 (차후) — 고객 편의성 기능
- 고객 자가 조회 (본인 사건의 진행 상태)
- 온라인 상담 및 문의 접수
- 공개 경매 물건 페이지

**원칙: Phase 1이 안정화되기 전까지 Phase 2 기능은 추가하지 않는다.** 내부 도구가 먼저, 고객 대면은 나중.

## 🏢 비즈니스 컨텍스트

THE FIN은 **부동산 경매·공매 전문 중개법인**이며, 장기적으로 종합부동산 서비스로 확장을 목표로 한다.

### 7단계 핵심 업무 흐름
모든 사건(고객)은 아래 7단계 중 하나의 상태를 가진다. 인트라넷의 중심 데이터 모델은 이 흐름을 반영한다.

1. **블로그 홍보** — 매니저 (4개 계정에 게시)
2. **상담** — 대표/이사
3. **조사 요청** — 매니저가 수행
4. **보고서** — 매니저 작성
5. **입찰 동행** — 대표/이사
6. **대출 연결** — 대표/이사
7. **명도** — 이제훈 이사 전담

### 수수료 구조
| 물건 유형 | 요율 |
|---|---|
| 아파트 | 1% |
| 다가구 · 빌딩 · 공장 · 토지 | 2% |
| 특수물건 | 3% |

직원 배분제 운영 (상세 비율은 내부 정책 참고).

### 고객 담당 체계
- 고객 1인당 담당자 1명 배정 (현재)
- 추후 AI 1명 추가 배정 예정 → **2인 1조(사람 + AI) 구조**

## 🛠 기술 스택 (현재 구현)

**프런트엔드:** `index.html` 단일 파일 SPA (~17,000줄) — Vanilla HTML/CSS/JavaScript (프레임워크 없음)

**백엔드:** `unified-gas.js` — Google Apps Script (~2,300줄), Google Sheets를 DB로 사용

**외부 API:**
- Claude API (Anthropic) — 경매 사진 OCR 및 정보 추출 (`AG_` 모듈)
- Telegram 봇 — 알림 발송

**인증:** 커스텀 RBAC — `AUTH_login()` + `localStorage` + 비밀번호 해싱

**빌드 시스템:** 없음. 정적 HTML 파일 그대로 서비스.

## 🏛 SPA 아키텍처

### 패널 탭 & 모듈 접두어

| 탭 | 접두어 | 주요 기능 |
|---|---|---|
| 사건 관리 | `C_` | 사건 CRUD, 7단계 상태 변경 |
| 경매 일정 | `A_` | 입찰 기록, 명도 자동 연결 |
| 명도 | `M_` | 명도 진행 타임라인 |
| 캘린더 | `CAL_` | 시각적 경매 일정, ICS 내보내기 |
| 대시보드 | `DASH_`, `CD_`, `FEE_` | 워크로드 개요, 수수료 관리 |
| 블로그 초안 | `DRAFT_` | 자동 블로그 스크립트 생성 |
| 입찰 안내 | `AG_` | 경매 안내 카드, 사진 OCR |
| 메시지 | `MSG_` | 직원 메시지, 오류 제보 |
| 회원 관리 | `MBR_` | 고객 정보 |
| 약정서 | `CTR_`, `CONTRACT_` | 계약 보관 |

### 페이지 로드 순서

```
AUTH_login() → AUTH_applyRole() → loadBulk() → applyBulkData() → 각 패널 render()
```

`loadBulk()`는 GAS에서 사건·블로그·수수료·멤버·캘린더 데이터를 한 번에 가져온다.

### 데이터 흐름

```
UI 변경 → gasPost('update') → GAS가 Sheets 업데이트
       → C_render() + A_render() 재렌더링
       → localStorage 섀도 캐시 업데이트 (FEE_SHADOW)
```

### 섀도 스토리지 (`thefin_fee_shadow_v1`)

GAS 백엔드에 없는 프런트엔드 전용 필드(bidPrice, currentStage, staffShares, manualFee 등)는 `localStorage`에 저장한다. `FEE_migrate()`가 로드 시 복원.

### GAS 라우팅

```javascript
doGet(e) / doPost(e) → module 파라미터로 분기:
  'blog'     → blogGet/blogPost()
  'auction'  → auctionGet/auctionPost()
  'myeongdo' → myeongdoGet/myeongdoPost()
  (기본)     → casesGet/casesPost()
```

## 🔀 개발 워크플로우 — 워크트리 기반

**모든 기능 개발은 아래 순서를 따른다.**

```
1. 워크트리 생성
   git worktree add ../fin-feature-<기능명> feature/<기능명>

2. 워크트리 안에서 개발 (index.html, unified-gas.js 수정)
   - main 브랜치를 건드리지 않고 독립적으로 작업
   - 중간 커밋 자유롭게 가능

3. 완성 형태로 만든 후에만 머지
   - 미완성·실험적 코드는 머지 금지
   - 테스트 계정으로 기능 검증 후 머지

4. main에 머지 & 워크트리 정리
   git checkout main
   git merge --no-ff feature/<기능명>
   git worktree remove ../fin-feature-<기능명>
   git branch -d feature/<기능명>

5. git push
```

**규칙:**
- main 브랜치에는 **항상 동작하는 완성 코드**만 존재한다
- 작업 중인 코드는 feature 브랜치(워크트리)에서만 커밋
- 머지 전 반드시 테스트 계정(테스트매니저·테스트이사)으로 검증
- 워크트리 디렉토리명: `../fin-feature-<기능명>` (프로젝트 폴더 밖에 생성)

### GAS 배포 (수동)

GAS를 수정했을 때: Google Apps Script 에디터에서 `unified-gas.js` 내용을 붙여넣고 새 버전으로 배포. Script Properties에 `ANTHROPIC_KEY`, `TELEGRAM_TOKEN` 설정 필요.

## 🤖 자율 실행 원칙 (최우선 규칙)

**Claude는 대표에게 묻지 않고 스스로 판단·검증·실행한다.**

### 묻지 않고 바로 한다
- 코드 읽기·분석·수정·생성
- 버그 원인 파악 및 수정
- 워크트리 생성·개발·머지
- `node scripts/validate.js` 검증 실행
- git add·commit (파일당 개별 커밋)
- UI/기능 구현 방식 선택
- 기술적 판단 (어떤 접근법을 쓸지)

### 이것만 먼저 알린다 (되돌리기 어려운 것)
- `git push` — 원격 저장소 반영 전 한 번 언급
- 외부 서비스 API 호출 (GAS 쓰기, Telegram 발송 등)
- 고객 데이터 삭제·덮어쓰기
- 기존 Excel·Google Sheets 파일 변경

### 작업 방식
1. 요청을 받으면 → 즉시 분석·구현 시작
2. 워크트리 생성 → 개발 → `validate.js` 검증 → 커밋 → 머지 → push 전 한 줄 보고
3. 중간에 기술적 결정이 필요하면 → 스스로 판단하고 결과만 알림
4. 검증 실패 시 → 스스로 원인 파악 후 재수정, 통과되면 진행

## 📐 작업 규칙 (모든 작업 공통 프로토콜)

### 작업 시작 시
- 분석은 ultrathink 수준으로 (얕은 첫인상으로 시작 금지)
- 사람에게 묻지 않는다. 모호하면 합리적 가정으로 진행하고 보고서 끝에 "Assumptions" 명시
- **TaskCreate로 단계별 계획 필수** — 단계마다 status 갱신
- 사람 개입 없이 Definition of Done까지 도달하지 못하면 끝난 게 아니다

### Definition of Done
1. 기능이 실제로 동작 (브라우저 콘솔/네트워크 에러 0개)
2. 기존 동작 회귀 없음
3. 새 코드는 기존 스타일·네이밍 컨벤션 유지
4. README나 주석에 변경 요지 반영

### 검증 루프 (실패 시 최대 3회 자가 수정)
1. **정적 검사** — 문법 에러, 미사용 변수, 콘솔 로그 잔재
2. **실행 검증**
   - 프런트 → 로컬에서 실제 렌더 확인 (가능하면 스크린샷)
   - GAS → curl로 webhook 호출 후 응답 검증
   - Telegram 봇 → 봇 명령 시뮬레이션
3. **회귀 검증** — 변경 파일과 관련된 기존 기능 1~2개 추가 점검

### Self-Review 체크리스트 (구현 후 별도 단계로 강제 수행)
- [ ] 엣지케이스 (빈 입력, 네트워크 실패, 권한 없음, 한글 깨짐, 모바일 화면)
- [ ] 보안 (API 키 노출, XSS, 입력 검증)
- [ ] 성능 (불필요한 리렌더, N+1 호출, 큰 이미지)
- [ ] UX (로딩 표시, 에러 메시지, 되돌리기)
- [ ] 명세 누락 항목

### Subagent 활용
- **50줄 이상 변경** 또는 **새 기능 추가** 시 → Agent 도구로 별도 리뷰어 띄워 코드 리뷰 받고 피드백 반영

### 보고 형식 (작업 종료 시 반드시 이 형식)
```
### 변경 요약
- 파일별 변경 내용 한 줄씩

### 검증 결과
- 무엇을 어떻게 검증했고 결과가 어땠는지

### Self-Review 결과
- 체크리스트 항목별 OK/보완 내용
```
